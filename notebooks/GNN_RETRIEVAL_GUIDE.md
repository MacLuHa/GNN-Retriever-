# GNN Retrieval: подробное объяснение

Этот документ объясняет, зачем в проекте нужна GNN, как она работает в вашем notebook и как ей пользоваться после обучения.

Связанный notebook:
- `notebooks/gnn_retrieval_neo4j.ipynb`

## 1. Зачем вообще нужна GNN в retrieval

В классическом поиске есть две основные стратегии:
1. Лексический поиск (`BM25`) — хорошо находит точные совпадения слов.
2. Семантический поиск (эмбеддинги текста) — хорошо ловит смысл.

Проблема юридического/регуляторного домена: релевантные куски часто связаны через сущности и контекст графа, а не только через прямое совпадение слов.

Примеры, где граф помогает:
1. В запросе упомянута сущность, а в нужном чанке есть только связанная сущность.
2. Норма описана в другом формулировании, но через связи `RELATED_TO` можно перейти к нужным узлам.
3. Нужно ранжировать не просто по словам, а по «структурной близости» в графе знаний.

Именно это даёт GNN:
- учитывает структуру `Chunk -> Entity -> Entity`,
- создаёт графовые эмбеддинги (`gnn_embedding`),
- добавляет графовый сигнал в итоговый `fusion score` retrieval.

## 2. На какой задаче обучается GNN в текущем notebook

В текущем notebook модель решает задачу `link prediction` для пары:
- вход: `Chunk` и `Entity`,
- цель: предсказать, есть ли ребро `(:Chunk)-[:MENTIONS]->(:Entity)`.

Почему это полезно для retrieval:
- если модель хорошо предсказывает релевантные `Entity` для `Chunk`,
- значит она научилась кодировать тематическую и структурную связь,
- и эти эмбеддинги можно использовать для ранжирования при поиске.

Важно: это retrieval proxy-задача. Для продового `query -> chunk` обычно добавляют отдельный query-encoder и fusion с BM25.

## 3. Как устроен граф в Neo4j

В вашем ТЗ и notebook используется схема:

1. Узлы:
- `(:Chunk {chunk_id, text, ...})`
- `(:Entity {entity_id, entity_name/value/normalized_value, ...})`

2. Рёбра:
- `(:Chunk)-[:MENTIONS]->(:Entity)`
- `(:Entity)-[:RELATED_TO]->(:Entity)`

Идея:
- `MENTIONS` задаёт прямую привязку доказательства (`Chunk`) к сущности.
- `RELATED_TO` расширяет контекст через соседние сущности.

## 4. Что происходит в notebook по шагам

## 4.1 Подключение и проверка схемы

Notebook подключается к Neo4j и проверяет:
- сколько `Chunk`,
- сколько `Entity`,
- сколько связей `MENTIONS`.

Это быстрый sanity-check, что данные действительно есть.

## 4.2 Выгрузка данных из Neo4j

Загружаются 4 таблицы:
1. `chunks_df`
2. `entities_df`
3. `mentions_df`
4. `related_df`

Для `Entity` используется fallback по имени:
- `entity_name` -> `normalized_value` -> `value`.

Это сделано, чтобы notebook не ломался, даже если в базе поля названы по-разному.

## 4.3 Фичи узлов

Сейчас используется `HashingVectorizer`:
- из текста `Chunk` строится вектор `chunk_x`,
- из текста `Entity` строится вектор `entity_x`.

Это baseline-решение:
- просто и быстро,
- не требует скачивания тяжёлых моделей,
- подходит для запуска «здесь и сейчас».

Минус:
- качество обычно ниже, чем у современных transformer-эмбеддингов.

## 4.4 Сборка графа для PyTorch Geometric

Создаётся `HeteroData` с двумя типами узлов (`chunk`, `entity`) и четырьмя типами рёбер:
1. `chunk -mentions-> entity`
2. `entity -mentioned_in-> chunk` (обратное)
3. `entity -related_to-> entity`
4. `entity -related_to_rev-> entity` (обратное)

Зачем обратные рёбра:
- сообщение в GNN должно проходить в обе стороны,
- это улучшает обучение и устойчивость.

## 4.5 Train/Val/Test split

Положительные примеры — реальные `MENTIONS` ребра.

Сплит:
1. `train` = 80%
2. `val` = 10%
3. `test` = 10%

Отрицательные примеры (`negative sampling`):
- для каждого `chunk` подбирается случайный `entity`,
- но только если такого ребра нет в графе.

## 4.6 Модель GraphSAGE

Используется гетерогенный GraphSAGE из 2 слоёв:
1. `HeteroConv(..., hidden_dim)`
2. `HeteroConv(..., out_dim)`

После второго слоя эмбеддинги L2-нормализуются.

Скор пары `(chunk, entity)`:
- скалярное произведение эмбеддингов.

Loss:
- `BCEWithLogits` на позитивных и негативных парах.

## 4.7 Метрики качества

В notebook считаются:
1. `Recall@K` — попала ли правильная сущность в top-K.
2. `MRR` — насколько высоко в ранжировании стоит правильная сущность.

Эти метрики показывают пригодность эмбеддингов для retrieval-задач.

## 4.8 Сохранение результата

После обучения notebook пишет в Neo4j:
1. `Chunk.gnn_embedding`
2. `Entity.gnn_embedding`
3. `gnn_model_version`
4. `gnn_updated_at`

То есть база начинает хранить графовые эмбеддинги, готовые для онлайн-поиска.

## 5. Как пользоваться моделью после обучения

Ниже практический online pipeline.

## 5.1 Шаг A: получить вектор запроса

Нужен `query_embedding` в том же векторном пространстве, что и `Entity.gnn_embedding`.

Варианты:
1. Базовый: тот же векторизатор + простой projection head.
2. Продовый: отдельный query-encoder, дообученный на парах `query -> entity`.

Если пространство будет другое, ранжирование станет нестабильным.

## 5.2 Шаг B: найти top-K сущностей

Считаем схожесть `query_embedding` с `Entity.gnn_embedding`:
- cosine similarity или dot product.

Результат: список `top_entity_ids`.

## 5.3 Шаг C: перейти от сущностей к evidence чанкам

Через граф:
- `(:Chunk)-[:MENTIONS]->(:Entity)`

То есть находим чанки, которые упоминают top-сущности.

## 5.4 Шаг D: объединить с BM25 и semantic

По ТЗ итоговый скор:

`S = 0.45 * norm(BM25) + 0.35 * norm(GNN) + 0.20 * norm(semantic_similarity)`

После этого:
1. dedup-aware merge,
2. ограничение diversity (например, не более N чанков на документ),
3. возврат цитат (`doc_id/page/span`).

## 5.5 Минимальный Cypher для entity->chunk

```cypher
MATCH (c:Chunk)-[:MENTIONS]->(e:Entity)
WHERE e.entity_id IN $top_entity_ids
RETURN
  c.chunk_id AS chunk_id,
  c.doc_id AS doc_id,
  c.page AS page,
  c.span_start AS span_start,
  c.span_end AS span_end,
  collect(e.entity_id) AS matched_entities
```

## 6. Как понять, что модель обучилась нормально

Минимальные признаки, что всё в порядке:
1. `loss` на train убывает.
2. `Recall@10` и/или `MRR` на val/test лучше случайного baseline.
3. Top-entity результаты выглядят содержательно при ручной проверке нескольких чанков.

Красные флаги:
1. Метрики не растут вообще.
2. Все предсказания однотипные (коллапс эмбеддингов).
3. Слишком мало ребер `MENTIONS` для стабильного обучения.

## 7. Типичные ограничения текущего notebook

1. Используется `HashingVectorizer` (baseline), без языковой модели.
2. Negative sampling — случайный, без hard negatives.
3. Нет отдельного обучения query encoder.
4. Оценка только на link prediction proxy, а не на полном e2e retrieval benchmark.

Это нормально для MVP и первого рабочего контура.

## 8. Что улучшать следующим шагом

1. Заменить baseline-фичи на transformer-эмбеддинги (`Chunk`/`Entity`).
2. Добавить hard negative sampling (близкие, но нерелевантные сущности).
3. Дообучить отдельный `query -> entity` энкодер.
4. Добавить полноценную offline-оценку retrieval (`Recall@K` по query->chunk).
5. Автоматизировать переобучение и версионирование моделей.

## 9. Коротко: в чём практическая ценность

GNN в вашем проекте нужна, чтобы retrieval учитывал не только текст, но и структуру знаний.

Результат:
1. лучшее покрытие релевантных документов,
2. более устойчивый поиск в случаях, где слова не совпадают напрямую,
3. explainable-путь через сущности и рёбра графа,
4. готовый графовый сигнал для hybrid scoring вместе с BM25 и semantic similarity.
