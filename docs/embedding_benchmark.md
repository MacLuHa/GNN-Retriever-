# Embedding Benchmark: baseline vs gnn

Скрипт сравнивает качество retrieval для:
- baseline векторов из Qdrant (`Chunk` и `Entity` по `embedding_id`),
- `gnn_embedding` из Neo4j (`Chunk.gnn_embedding` и `Entity.gnn_embedding`).

## Что делает

1. Создаёт (или переиспользует) golden dataset из `(:Chunk)-[:MENTIONS]->(:Entity)`.
2. Строит общий кандидатный пул сущностей, доступных в обоих пространствах.
3. Считает на одинаковом наборе запросов метрики:
- `Precision@K`
- `Recall@K`
- `Hit@K`
- `MRR@K`
- `nDCG@K`

## Что означают метрики

Ниже `query` = один `chunk_id`, а `relevant` = `positive_entity_ids` из golden dataset.

### Precision@K

Доля релевантных сущностей среди первых `K` результатов.

Формула:
`Precision@K = (# релевантных в топ-K) / K`

Интерпретация:
- хорошо показывает "чистоту" верхушки выдачи;
- полезно, когда пользователю важны именно первые ответы.

### Recall@K

Какую долю всех правильных сущностей для запроса удалось покрыть в топ-K.

Формула:
`Recall@K = (# релевантных в топ-K) / (# всех релевантных для query)`

Интерпретация:
- показывает полноту;
- важна, когда нужно не пропускать связанные сущности.

### Hit@K

Бинарная метрика "попали или нет" в топ-K хотя бы в одну релевантную сущность.

Формула:
- `1`, если в топ-K есть хотя бы один релевантный результат;
- `0`, если нет.

Интерпретация:
- простая и устойчивая;
- не учитывает качество ранжирования внутри топ-K, только факт попадания.

### MRR@K (Mean Reciprocal Rank at K)

Оценивает, насколько рано появился первый релевантный результат.

Для одного запроса:
- если первый релевантный на позиции `r` и `r <= K`, то `RR = 1/r`;
- иначе `RR = 0`.

`MRR@K` = среднее `RR` по всем запросам.

Интерпретация:
- сильнее штрафует случаи, когда правильный ответ появляется поздно;
- хороша для сценариев, где важен "первый хороший ответ".

### nDCG@K

Нормализованная метрика качества ранжирования с большим весом верхних позиций.

В бенчмарке используется бинарная релевантность (релевантно/нерелевантно):
- `DCG@K = sum(1 / log2(rank + 1))` по релевантным в топ-K;
- `IDCG@K` — идеальный `DCG@K` (лучший возможный порядок);
- `nDCG@K = DCG@K / IDCG@K`.

Интерпретация:
- учитывает и полноту, и порядок;
- значение всегда в диапазоне `[0, 1]`;
- `1.0` означает идеальный порядок в пределах `K`.

## Как агрегируются метрики в отчете

Для каждого `chunk_id` сначала считаются query-level метрики.  
Дальше в `summary.json` и `summary.csv` сохраняется среднее по всем query:
- `baseline` — средние метрики для Qdrant-векторов,
- `gnn` — средние метрики для `gnn_embedding`,
- `delta_gnn_minus_baseline` — разница `gnn - baseline` по каждой метрике.

Положительный `delta` означает, что GNN лучше по данной метрике.

## На что смотреть при сравнении baseline и gnn

1. Если важны первые результаты:
- приоритет `MRR@K`, `Precision@K`, `nDCG@K`.

2. Если важна полнота:
- приоритет `Recall@K` и `Hit@K`.

3. Практическая диагностика:
- `Recall@K` растет, а `MRR@K` падает: модель находит больше правильных, но ранжирует их ниже;
- `Hit@K` растет при слабом росте `Precision@K`: чаще есть хотя бы одно попадание, но шум в топе остается;
- рост `nDCG@K` обычно означает реальное улучшение порядка выдачи.

## Запуск

```bash
python scripts/run_embedding_benchmark.py \
  --neo4j-uri "$NEO4J_URI" \
  --neo4j-user "$NEO4J_USER" \
  --neo4j-password "$NEO4J_PASSWORD" \
  --qdrant-url "$QDRANT_URL" \
  --qdrant-api-key "$QDRANT_API_KEY" \
  --qdrant-chunk-collection "$QDRANT_CHUNK_COLLECTION" \
  --qdrant-entity-collection "$QDRANT_ENTITY_COLLECTION" \
  --qdrant-id-field "embedding_id" \
  --golden-mode auto \
  --golden-path benchmarks/golden_dataset.jsonl \
  --golden-limit 2000 \
  --ks 1 5 10 20
```

## Артефакты

По умолчанию сохраняется в `benchmarks/runs/<timestamp>/`:
- `summary.json` — итоговые метрики baseline/gnn и `delta_gnn_minus_baseline`
- `summary.csv` — сводная таблица
- `per_query_metrics.csv` — метрики по каждому `chunk_id`

Golden dataset сохраняется в `benchmarks/golden_dataset.jsonl`.
